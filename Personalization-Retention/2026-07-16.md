# 推荐与留存日报 — 2026-07-16

## 今日头条：ICLR 2026 论文：双边匹配不该只追「匹配数最大化」或「公平」，而应直接优化「留存」

在线约会、招聘等双边平台的推荐算法通常以最大化 total matches 为目标，但这会造成严重失衡：少数热门用户收到海量匹配，而大量用户几乎无人问津、最终流失。这篇 ICLR 2026 论文《Beyond Match Maximization and Fairness》指出，用 fairness（如均等曝光）来救场也不够——用户并不会因为「曝光被拉平」就回报平台，把留存交给公平约束等于交给运气。作者首次把「最大化用户留存」形式化为双边匹配的独立优化目标，直接对准订阅制平台真正在乎的指标。

「为什么重要」：CMB 的商业模式高度依赖订阅与长期活跃，这篇把「留存」而非「当日匹配数/公平」设为推荐优化目标的框架，正好对应我们真正的北极星，值得产品与算法团队一起读。

**生成式推荐的 Scaling 之争：直接把 LLM 当推荐器，可比 Semantic ID 方案再涨 20%**

关于 generative recommendation 的两条技术路线正在收敛出结论。论文《Understanding Generative Recommendation with Semantic IDs from a Model-scaling View》重新审视了「LLM-as-RS」范式——即不再把物品编码成 Semantic ID（SID）再解码，而是直接用大模型做推荐。实验显示该范式具备更优的 model scaling 特性：随参数规模增长，其效果最高可比 SID-based GR 的最优可达性能再提升约 20%。这提示业界：生成式推荐的天花板可能不在 tokenization 技巧，而在模型规模与语义表示本身。

「为什么重要」：约会场景的候选池是「人」，item 语义极其稀疏且个性化，SID 方案的信息损失更明显——这条 scaling 证据帮我们判断在 reciprocal 召回/排序上是否值得押注更大的生成式模型。

[原文链接](https://arxiv.org/abs/2509.25522)

**Spotify 工程实践：为什么把「个性化」与「实验」拆成两套技术栈**

Spotify 在服务 5 亿用户时坚持让 personalization stack 与 experimentation stack 各司其职：ML 栈负责「产出推荐」，实验栈负责「评估推荐」。原因是个性化系统有一批实验工具天然装不下的硬性要求——需要多样模型类型（neural nets、boosting、bandits）、丰富特征、低延迟 inference 和实时数据采集；而实验平台要保证多团队能并行、大规模地跑 A/B 而互不干扰。清晰的关注点分离让两侧都能按各自最优目标演进，产品团队扩展时摩擦更小。

「为什么重要」：CMB 的推荐迭代同样受制于「实验能不能快、能不能多队并行」，这套把服务与评估解耦的架构思路，是我们扩大 A/B 吞吐、避免实验互相污染的直接参考。

[原文链接](https://engineering.atspotify.com/2026/1/why-we-use-separate-tech-stacks-for-personalization-and-experimentation)
